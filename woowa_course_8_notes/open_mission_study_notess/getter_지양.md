# 📖 지식 정리 (우테코를 하며 학습한 내용)
## 1️⃣ `getter`를 사용하는 대신 객체에 메시지를 보내자
``` text
객체지향 프로그래밍은 객체가 스스로 일을 하도록 하는 프로그래밍이다.

상태를 가지는 객체를 추가했다면 객체가 제대로 된 역할을 하도록 구현해야 한다.
객체가 로직을 구현하도록 해야한다.
상태 데이터를 꺼내 로직을 처리하도록 구현하지 말고 객체에 메시지를 보내 일을 하도록 리팩토링한다.
```

---

## ☑️ 고민한 내용
**Lotto 객체가 값을 꺼내지 않고 스스로 당첨 번호와 비교하도록 개선**하여 객체 간 불필요한 결합을 줄이고,
캡슐화와 객체 중심 설계를 강화했다. 이 과정에서 getter 사용도 자연스럽게 줄어들었다.

`LottoTickets`가 데이터를 외부로 단순 전달하는 형태에서
- toDisplayXxx() 기반의 `List<String>`을 반환하여 출력 레이어와의 결합도를 줄임. (`LottoResult` 또한 마찬가지)

Winning 객체가 내부 데이터를 꺼내서 처리하던 방식에서
- **→ “데이터를 꺼내지 말고 메시지를 보내자” 원칙 적용**

`BonusNumber`와 `WinningNumbers` 내부에서 중복 검사까지 수행하도록 변경하여
- → 외부에서 불필요한 데이터 접근 제거.

**getter 지양과 현실적인 타협**
- 가능하면 데이터를 꺼내지 않는 방향(객체 협력 중심)으로 개선.
- 하지만 **당첨 통계 계산 과정은 티켓, 당첨 번호, 보너스 번호 간 강한 의존성** 존재.
- 이 경우에는 완전한 getter 제거가 어렵다고 판단하여
    - **→ 필요한 데이터는 꺼내서 비교하도록 현실적으로 결정.**
