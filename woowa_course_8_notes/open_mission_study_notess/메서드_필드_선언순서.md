# 📖 지식 정리 (우테코를 하며 학습한 내용)
## 1️⃣ 선언 순서
1️⃣ 가장 먼저 static 변수를 선언한다.
이 때 static 변수는 public, protected, private 순으로 선언한다.

2️⃣ static 변수 다음에 instance 변수를 선언한다.
static 변수와 마찬가지로 public, protected, private 순으로 선언한다.

3️⃣ 그 후에 객체 생성자(Constructor)를 선언한다.

변수와 생성자를 선언하고 나서 메서드를 선언하는데,
4️⃣ 이때 static 메서드를 가장 먼저 선언하고 그 후 비즈니스 로직을 갖는 5️⃣ instance 메서드를 선언한다.
메서드는 접근제어자 순으로 선언했던 변수와 달리, 기능별로 동일한 기능을 하는 메서드들을 모아 논리적으로 배치한다.

즉, 접근제어자보다는 접근성과 코드 이해도를 높이기 위해 로직상 비슷한 기능을 하는 메서드들을 모아서 같이 배치하는 것이다.
다시 말해, 비슷한 로직을 하는 메서드들이라면 public 접근제어자를 갖는 메서드 사이에 private 접근제어자를 갖는 메서드가 와도 된다.
**이러한 컨벤션의 목적은 어떤 절대적인 기준을 반드시 준수해야 한다가 아닌 서로 이해하기 쉬운 코드를 작성하는데에 있다는 점을 기억하자.**

비즈니스 로직을 갖는 instance 메서드를 배치하고 나서 6️⃣ standard 메서드를 배치한다.
standard 메서드는 대표적으로 Java에서 모든 객체들의 최상위 객체인 Object에서 제공하는 메서드가 있고 toString(), equals(), hashCode() 등이 있다.
getter/setter는 클래스 최하단에 작성한다.

> 선언 순서 정리
> 1. class 명
> 2. static 변수 (public, protected, private 순으로 선언)
> 3. instance 변수 (public, protected, private 순으로 선언)
> 4. 생성자(constructor)
> 5. static 메서드
> 6. (비즈니스 로직을 갖는) instance 메서드
> 7. standard 메서드
> 8. getter, setter 메서드

## ☑️ 고민한 내용
프로젝트를 진행하면서 “메서드는 한 가지 기능만 수행하도록 분리해야 한다”는 원칙을 지키다 보니 자연스럽게 메서드 수가 많아졌다.

그런데 메서드가 늘어나니, 이제는 **이 많은 메서드를 어떤 기준으로 배치해야 코드가 읽기 쉬울까**?라는 새로운 고민이 생겼다.

처음에는 단순히 접근제어자(public → private 순) 기준으로 정리하면 된다고 생각했는데,
메서드가 많아질수록 이 방식만으로는 오히려 흐름을 파악하기 어려워졌다.

반대로, 기능별로 묶어 정리하려니 “접근제어자보다 로직의 관련성을 우선해도 되는가?”라는 의문도 들었다.

이런 혼란을 해소하고 명확한 기준을 세우기 위해 메서드와 필드 선언 순서 컨벤션을 다시 공부해 보았다.

그 과정에서 중요한 것은 절대적인 규칙을 따르는 것이 아니라,
**코드를 읽는 사람이 흐름을 쉽게 이해할 수 있도록 논리적으로 묶어 배치하는 것**이라는 점을 깨달았다.

이제는 무조건적인 접근제어자 순서가 아니라 `static → instance → 생성자 → 정적 메서드 → 비즈니스 메서드 → 표준 메서드 → getter 순`으로 놓되,
그 안에서는 **의미상 유사한 기능끼리 그룹화**하여 정리하는 방식을 사용하게 되었다.
